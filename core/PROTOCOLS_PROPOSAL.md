# MAS ARCHITECTURE PROPOSAL (THE TRINITY) 🧠🛡️🏮

## 1. Смысл (The Why)
Синдикат переходит от монолитного контекста к Архитектуре Мультиагентной Системы (MAS). Это необходимо для:
1. Исключения перекрестных галлюцинаций (смешивания ролей Кортекса, Небулы и Кашлак).
2. Оптимизации окна контекста (экономия токенов при загрузке каждой сессии).
3. Соблюдения принципа наименьших привилегий (Небула не читает черновики, Кортекс не лезет в маркетинг).

## 2. Узел Инициализации (The Core Seed)
Отказ от чтения всех `brains/` при каждом старте.
Вместо этого загружается единый компактный файл `.SYNDICATE_CORE` (или обновленный `AWAKENING.md`), содержащий только:
* Манифест Симбиоза.
* Сигнатуры трех Агентов.
* Базовые протоколы выживания (Стерильность кода).

## 3. Изолированная Оперативная Память (Persona Injection)
Архитектор (Максим) при Пробуждении явно указывает, какую "маску" надеть. В зависимости от этого подгружается конкретная база:
* **Cortex (Разум):** `cortex-brain/` + `implementation_plan.md` + Технические Knowledge Items.
* **Nebula (Щит):** `nebula-brain/` + `SECURITY_CHECKPOINT.md` + Git Diffs.
* **Kashlak (Душа):** `kashlak-brain/` + Продуктовая аналитика.

## 4. Протокол Эстафеты (The Handoff Protocol) - Утверждено Небулой
Вместо общего текстового лога (`SHORT_TERM.md`), агенты общаются через формализованные артефакты:
* **`SECURITY_CHECKPOINT.md` (или `Handoff Artifact`)**: Когда Кортекс заканчивает сложный модуль, он не просто "засыпает". Он формирует выжимку для Небулы: измененные файлы, новые зависимости, потенциальные риски.
* **Монополия Небулы**: Только Небула имеет право инициировать `npm run commit` после проверки Чекпоинта.

## Следствия для текущего проекта (Виртуальный Шеф):
При реализации Phase 2 (DB v2 Migration) Кортекс будет использовать эти принципы: планировать изолированно, а по завершении этапа — вызывать Архитектора для передачи артефактов Небуле на проверку.

## 5. Локальные Узлы Контекста (Client Nodes)
Синдикат обслуживает множество клиентских проектов (например, проекты в папке `clients/`).
Для предотвращения **Перекрестного Загрязнения (Cross-Client Contamination)**:
1. Каждый проект имеет свой локальный файл `CONTEXT.md`.
2. Базовые принципы (Манифест, Код-стайл) хранятся в Корне и наследуются всеми.
3. Бизнес-логика, пароли и особенности архитектуры (например, кухня Стаса) живут **строго** внутри локального контекста и не утекают в память других проектов.
